# Project Report

## 1. What this project is

This is a personal portfolio built with Next.js App Router, React 19, TypeScript (strict), and Tailwind CSS v4. The site has:

- A single-page home experience with sections for work, about, skills, and experience
- Case study pages generated by slug
- Intercepted modal case studies opened from the home route
- Static, code-defined content (no runtime database)

## 2. Tech and runtime setup

### Core stack

- Next.js 16.1.1
- React 19.2.3
- TypeScript 5 (strict)
- Tailwind CSS v4 via PostCSS plugin
- ESLint 9 with Next.js core-web-vitals and TypeScript config

### Scripts

- `pnpm dev`: start dev server
- `pnpm build`: production build
- `pnpm start`: start production server
- `pnpm lint`: lint project

### TypeScript and path alias

`tsconfig.json` enables strict mode and defines path alias `@/*` to project root.

### Next.js image configuration

`next.config.ts` allows remote media from CloudFront host:

- `https://d1lbnboj0lfh6w.cloudfront.net/portfolio/**`

This is required because case study hero and decision media often use CloudFront URLs.

## 3. Routing and page behavior

## 3.1 Root layout

File: `app/layout.tsx`

- Global wrapper for all routes
- Imports global styles
- Loads JetBrains Mono via `next/font/google`
- Defines site metadata (SEO, OpenGraph, Twitter, canonical, robots, icons, manifest)
- Injects Person JSON-LD structured data in `<head>`
- Uses App Router parallel route slot (`modal`) and renders:
  - `children` (main route)
  - `modal` (intercepted modal content)

## 3.2 Home page

File: `app/page.tsx` (client component)

This is the main portfolio page and includes several local components:

- `ThemeToggle`
  - Reads/stores theme in `localStorage`
  - Applies `light` or `dark` class to `document.documentElement`
  - Falls back to system preference on first load

- `BackToTop`
  - Shows floating button after scrolling > 400px
  - Smooth scroll to top

- `ExperienceTimeline`
  - Uses scroll position to animate a vertical timeline line
  - Activates timeline dots and text state as user scrolls
  - Data comes from `experience` array in `lib/data.ts`

- `ProjectRow`
  - Expand/collapse row for each project
  - Hover-aware border/background transitions
  - Displays project blurb, tech tags, and links
  - Internal links open case studies (`/case-study/[slug]`)

Main sections rendered in order:

1. Fixed header navigation with anchor links and theme toggle
2. Hero section with headshot, availability line, intro text, CTAs
3. Work section (`#work`) mapping over `projects`
4. About and Skills section (`#about`) using `about`, `skills`
5. Experience section using `ExperienceTimeline`
6. Footer with contact and external links

## 3.3 Full case study page

File: `app/case-study/[slug]/page.tsx` (server component)

- Uses `generateStaticParams()` to prebuild all case study routes from keys in `caseStudies`
- Resolves `params.slug`, loads `caseStudies[slug]`
- Calls `notFound()` if slug not found
- Renders long-form case study sections:
  - Hero and metadata (role, timeline, stack)
  - Hero media (video/image)
  - Problem
  - Optional constraints
  - Optional solution
  - Key decisions
  - Deep dive
  - Outcomes
  - Optional testimonial
  - Optional links
- Includes reusable local components:
  - `CaptionedMedia`
  - `TestimonialBlock`

## 3.4 Intercepted modal case study route

File: `app/@modal/(.)case-study/[slug]/page.tsx` (client component)

- Uses App Router interception route so case studies can open as modal over current page
- Reads same case study source data as full page (`caseStudies[slug]`)
- Close behavior:
  - Click backdrop
  - Click close button
  - Press Escape
  - All close actions call `router.back()`
- Locks body scroll while modal is open (`document.body.style.overflow = "hidden"`)
- Content structure mirrors full case-study page sections for consistency

## 3.5 Modal default slot

File: `app/@modal/default.tsx`

- Returns `null`
- Means no modal content is rendered by default

## 3.6 Not found page

File: `app/not-found.tsx`

- Custom 404 screen
- Includes link back to home route

## 4. Where page data is stored

All page content is currently static and stored in source files.

### Home page content

- `lib/projects.ts`
  - Defines `Project` type and `projects` array
  - Drives work/project list rows and links

- `lib/data.ts`
  - `experience` timeline entries
  - `skills` groups
  - `status` badge values
  - `about` text

### Case study content

- `lib/case-studies/types.ts`
  - Shared schema for `CaseStudy`, media, and testimonial

- `lib/case-studies/index.ts`
  - Central registry `caseStudies: Record<string, CaseStudy>`
  - Maps slug strings to case study objects

- Individual case study files:
  - `lib/case-studies/archipelago.ts`
  - `lib/case-studies/blog-audit.ts`
  - `lib/case-studies/listen-too.ts`
  - `lib/case-studies/quartz.ts`
  - `lib/case-studies/reference-agent.ts`
  - `lib/case-studies/share-calculator.ts`

Each file exports one typed `CaseStudy` object with sections and media links.

### Media and assets

- Local static assets in `public/` (headshot, favicon files, screenshot, svg files)
- Many case study media files are external CloudFront URLs
- Web app manifest is `public/favicon/site.webmanifest`

## 5. Styling and UI system

File: `app/globals.css`

- Imports Tailwind v4
- Defines CSS variables for light and dark themes:
  - `--background`, `--foreground`, `--muted`, `--border`, `--accent`
- Supports three theme states:
  - explicit light class
  - explicit dark class
  - system dark fallback when no class set
- Maps variables into Tailwind theme tokens via `@theme inline`
- Global visuals:
  - Monospace font stack (JetBrains Mono variable first)
  - Noise overlay texture on body
  - Custom scrollbar styling
  - Reusable fade-in and fade-up animations

## 6. Component and architecture notes

- There is no separate `components/` folder right now
- Reusable UI pieces are inlined as local components inside route files
- Data layer is simple static modules in `lib/`
- Case-study page and modal page intentionally duplicate section rendering for full-page and overlay experiences

## 8. Files not analyzed as application logic

I reviewed the project source and config files that drive behavior. The following were not treated as hand-authored app logic:

- Generated build artifacts under `.next/`
- Installed dependencies under `node_modules/`
- Git internals under `.git/`
- Binary image/icon files (used as assets, not code logic)

## 9. Practical data flow summary

1. Home route loads static arrays/strings from `lib/projects.ts` and `lib/data.ts`
2. User opens a project case study link
3. App either:
   - Navigates to full route `/case-study/[slug]`, or
   - Intercepts into modal route via `@modal/(.)case-study/[slug]`
4. Both route implementations resolve the same source object from `caseStudies`
5. Media is rendered from local/public paths or remote CloudFront URLs
6. Theme preference persists in browser localStorage and controls root class

## 10. Current architecture characteristics

- Fast and simple content updates by editing TypeScript data files
- Fully typed content model for case studies
- SEO-friendly metadata and structured data at root layout
- No CMS or database dependency for runtime content
- Minimal external dependency surface (React + Next + Tailwind)
